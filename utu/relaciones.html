Anatomía del acoplamiento en Java

Cada vez que una clase conoce a otra, nace una relación — y cada relación tiene un costo de acoplamiento distinto. Este recorrido diseca las cinco formas en que dos clases pueden vincularse en Java, de la más débil a la más fuerte.

Usa el mapa de la izquierda o el espectro de abajo para saltar directamente a la relación que quieras estudiar.

Acoplamiento débil Acoplamiento fuerte
Dependencia Asociación Agregación Composición Herencia

El espectro del acoplamiento

A medida que avanzamos de izquierda a derecha, el acoplamiento aumenta y la flexibilidad para modificar el código disminuye. En el extremo débil, una clase apenas nota si la otra cambia; en el extremo fuerte, un cambio en una clase puede romper directamente a la otra. La regla general de un buen diseño orientado a objetos es simple: usa siempre la relación más débil que resuelva el problema.

💡
Por qué importa: el acoplamiento no es bueno ni malo en sí mismo — es un costo. Cuanto más fuerte es la relación entre dos clases, más difícil es probarlas por separado, reutilizarlas o cambiarlas sin efectos secundarios.
01

Dependencia — Dependency

"usa-a" · uses-a

Una clase depende de otra cuando la utiliza únicamente como parámetro de un método, variable local o valor de retorno — pero nunca la guarda como atributo. Es el acoplamiento más débil y efímero: la relación solo existe mientras dura esa llamada.

GestorPedidos.java
public class GestorPedidos {
    public void procesar(Impresora imp) {
        imp.imprimir("Recibo");
    }
}
Diagrama UML
◍
Ciclo de vida: el objeto Impresora no vive dentro de GestorPedidos; se le "presta" únicamente mientras dura la ejecución del método procesar().
02

Asociación — Association

"tiene-a" · has-a (referencia)

Una clase mantiene una referencia permanente a otra como atributo. A diferencia de la dependencia, este vínculo sobrevive más allá de un solo método. Ambos objetos pueden existir de forma completamente independiente, y la relación puede ser unidireccional o bidireccional.

Profesor.java
public class Profesor {
    private Curso curso;

    public Profesor(Curso curso) {
        this.curso = curso;
    }
}
Diagrama UML
◍
Ciclo de vida: es independiente en ambos sentidos — si el Profesor deja de existir, el Curso sigue existiendo sin ningún problema (y viceversa).
03

Agregación — Aggregation

"tiene-un(a), parte de" · shared has-a

Forma especial de asociación que representa una relación todo-parte débil. Las partes pueden existir sin el todo, e incluso pueden compartirse entre varios "todos" al mismo tiempo. Se reconoce en UML por un diamante vacío del lado del todo.

Equipo.java
public class Equipo {
    private List<Jugador> jugadores;

    public Equipo(List<Jugador> jugadores) {
        this.jugadores = jugadores;
    }
}
Diagrama UML
◍
Ciclo de vida: un Jugador puede existir sin pertenecer a ningún Equipo (agente libre), y en teoría el mismo jugador podría pasar por varios equipos a lo largo de su carrera.
04

Composición — Composition

"está compuesto de" · exclusive has-a

Forma fuerte de agregación: la parte no puede existir sin el todo. El todo crea, controla y destruye a sus partes — la propiedad es exclusiva. En UML se representa con un diamante relleno.

Casa.java
public class Casa {
    private final Habitacion habitacion;

    public Casa() {
        this.habitacion = new Habitacion();
    }
}
Diagrama UML
◍
Ciclo de vida: Habitacion nace dentro del constructor de Casa y muere con ella — no tiene sentido que exista fuera de esa casa en particular.
05

Herencia — Inheritance

"es-un(a)" · is-a

El acoplamiento más fuerte de todos. Una subclase hereda atributos y métodos de una superclase mediante extends, formando una relación "es un/una". Cualquier cambio en la superclase repercute directamente — y a veces silenciosamente — en todas sus subclases.

Animal.java · Perro.java
public class Animal {
    public void comer() { /* ... */ }
}

public class Perro extends Animal {
    public void ladrar() { /* ... */ }
}
Diagrama UML
⚠
El problema de la clase base frágil: como Perro queda ligado a toda la implementación interna de Animal, un cambio aparentemente inocente en la superclase puede romper el comportamiento de la subclase sin que se toque una sola línea de Perro.

Elegir el acoplamiento correcto

No existe una relación "buena" universal — existe la relación adecuada para lo que tu clase necesita expresar. La guía práctica es preferir siempre el vínculo más débil que resuelva el problema.

Tres reglas para recordar

  • 01Favorece la composición sobre la herencia cuando solo necesites reutilizar comportamiento, no crear una jerarquía "es-un(a)" real.
  • 02Si una clase solo necesita usar a otra por un instante, no la guardes como atributo — pásala como parámetro y mantenla como dependencia.
  • 03Antes de guardar una referencia permanente, pregúntate si de verdad hay una relación todo-parte (agregación/composición) o simplemente una asociación entre iguales.
